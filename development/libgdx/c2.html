    [2018]

    Strongly recommended for berinners: - make full copy of project before any manipulation (if your code is not only testing scrap) , this allow safe time and health sometimes
    - try always use stable versions of IDE idea EAP and anst canary etc is bad way for beginner, because it is 99% romantic adventure for pro.
    - try to open with another IDE for example if idea fail then use anst. In 2018 this allow detect silent fail with idea , with default libGDX 1.9.8 project , because only anst show message about deprecation of minSdk and targetSdk declaration inside AndroidManifest.xml file of libGDX project, created uses released stable version of jdx-setup.jar official tool. Gradle (android section) file have settings for minSdk and targetSdk parameters.
    - delete AndroidManifest.xml tag with minSdk and targetSdk declaration android gradle have similar settings, and xml tag deprecated, but idea can not show you notification about this, which you can see inside anst
    - choose REMIND ME TOMORROW when anst , at first project opening, ask you once about stroungly recommended update for gradle plugin, because this is very good update, OR YOUR PROJECT WILL BE BROKEN without any fast recover actions
    - File -> invalidate Caches/Restart... inside anst or idea Process need some time (all will be cleaned and rebuilded) and, for example, can solve situation when, after recoding from Java to Kotlin, your correct project without errors fail in time of Run -> DesktopLauncher command, with error can't found main class etc .
    Funally you need recreate desktop runnable configuration, usual way from official libgdx site tutorial . That create runnable configuration use Run -> Edit Configurations... -> + -> Application

    - Tools -> Kotlin -> Configure Kotlin Plugin Updates inside anst or idea You can update, and see Kotlin version number. For example 1.3.10-release-IJ2018.2-1 , that use it later for compare/change with build.gradle project level file settings:

    - File -> Settings... -> Build, Execution, Deployment -> Compiler -> Kotlin Compiler change target JVM version from 1.6 to 1.8 . Theoretically, if this not break something, then it can allow some new language opportunities
